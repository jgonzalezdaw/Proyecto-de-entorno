Instalando GIT en Windows / Linux / Mac

Para instalar GIT en nuestro sistema basta con seguir los siguientes pasos.

En Windows: Instalar Git en Windows es muy fácil. El proyecto msysGit tiene uno de los procesos de instalación más sencillos. Simplemente descarga el archivo exe del instalador desde la página de GitHub, y ejecútalo:
Descargar instalador para Windows

Una vez instalado, tendrás tanto la versión de línea de comandos (incluido un cliente SSH que nos será útil más adelante) como la interfaz gráfica de usuario estándar.

Nota para el uso en Windows: Se debería usar Git con el shell proporcionado por msysGit (estilo Unix), lo cual permite usar las complejas líneas de comandos de este libro (ver "guía oficial" al pie de página). Si por cualquier razón se necesitara usar el shell nativo de Windows, la consola de línea de comandos "cmd", se han de usar las comillas dobles en vez de las simples (para parámetros que contengan espacios) y se deben poner entre comillas los parámetros terminándolos con el acento circunflejo (^) si están al final de la línea, ya que en Windows es uno de los símbolos de continuación.

En Linux: Si quieres instalar Git en Linux a través de un instalador binario, en general puedes hacerlo a través de la herramienta básica de gestión de paquetes que trae tu distribución.

Si estás en Fedora, puedes usar yum: ( yum install git-core ).

O si estás en una distribución basada en Debian como Ubuntu, prueba con apt-get: ( apt-get install git )

En Mac: Hay tres maneras fáciles de instalar Git en un Mac.
1º La primera más sencilla es usar el instalador gráfico de Git, que puedes descargar desde la página de SourceForge:
Descargar instalador para Mac

2º La segunda es instalar Git a través de MacPorts. Si tienes MacPorts instalado, instala Git con ( sudo port install git-core +svn +doc +bash_completion +gitweb ).

3º La tercera es Homebrew. Si ya tienes instalado Homebrew, instala Git con ( brew install git )

Configurando Git

Ahora que tienes Git instalado toca configurarlo, esto solo se debe hacer una sola vez.
Git dispone de una herramienta llamada GitConfig que te permite configurar algunas opciones de Git, el archivo puede estar en tres rutas distintas.

  1. /etc/gitconfig: Donde se guarda la información de todos los usuarios y repositorios.
  2. ~/.gitconfig: Donde se guarda su información de usuario.
  3. .git/config: Donde se guarda la información de un repositorio (cada commit sobrescribe el archivo).

En sistemas Windows, Git busca el archivo .gitconfig en el directorio $HOME (%USERPROFILE% in Windows’ environment), que es C:\Documents and Settings\$USER para la mayoría de usuarios, dependiendo de la versión ($USER es %USERNAME% en el entorno Windows). También busca en el directorio /etc/gitconfig, aunque esta ruta es relativa a la raíz MSys, que es donde quiera que decidieses instalar Git en tu sistema Windows cuando ejecutaste el instalador.

Tu identidad
Lo primero que debes hacer es establecer tu nombre de usuario y dirección de email. Es importante porque los cambios (commits) en Git usan esta información:

git config --global user.name "Nombre Apellidos"
git config --global user.email SuEmail@example.com

Repositorios

Los repositorios son el lugar de trabajo donde se guardan todos los datos con los que trabajamos, estos repositorios pueden tener miles de archivos. Ademas de ello, un repositorio permite a más de una persona trabajar en dicho repositorio directamente, normalmente por medio de las ramas o por medio del uso de fork. Los repositorios suelen ser en su mayoría de casos públicos (a la vista de cualquiera), los cuales puede descargar y colaborar en el proyecto si quiere.

Un repositorio se inicia desde la web de GitHub cual posteriormente podremos acceder desde el terminal o consola del sistema usando los comandos (init, add, commit, push) como los principales comandos para trabajar en el repositorio (ver comandos)

Commits

Los commits es la forma para preparar un envió de datos al repositorio. Solo podemos hacer un commit si se han realizado cambios en el proyecto o se ha vuelto a un estado anterior con un REBASE o un RESET (ver comandos o ver guías). Tras realizar los cambios necesarios a nuestro proyecto podremos hacer un commit el cual nos preparara el/os archivos del proyecto junto a un nombre y descripción para enviarlo al repositorio. Git le dará un ID al commit así como también situara el HEAD en dicho commit. Posteriormente a elección del usuario podrá enviar o no el commit al repositorio.

Ramificaciones en Git

Las ramas es una forma de separar varias partes del trabajo y evitar que puedan producirse fallos o errores por parte del usuario y evitar tocar la rama principal. Para entenderlo mejor hay que explicar como funciona el sistema de Git. Git utiliza unos marcadores para saber donde está situado el usuario con su último estado. La rama principal se llama (Origin/Master) de esta forma podemos saber cual es la rama primaria. El marcador (HEAD) indica donde estamos situados en ese momento, rama y commit en el que nos encontramos. La última etiqueta suele ser el nombre de la rama en la que nos encontramos. En caso de hacer un MERGE (juntar las ramas en la principal o en otra rama) nos aparecerá el nombre de ambas o todas las ramas que están en ese mismo punto. Hay que tener en cuenta que antes de unir las ramas hay que actualizar al último estado en que se encuentra la rama de la cual hicimos la copia ya que podríamos eliminar parte del proyecto que no tenemos al día. Para entenderlo mejor mira (ver guías)

Haciendo Fork

Qué es hacer un Fork?. Cuando entras en un repositorio de Git que no te pertenece puedes ver que en la esquina superior derecha hay un botón [FORK|0] (donde 0 habrá cualquier número). Al pulsar vas hacer una copia de ese repositorio a tu cuenta de Git y podrás trabajar como quieras en ese repositorio ya que tienes una copia exacta (Ten en cuenta que FORK no es hacer un COPY sino más bien es como crear una rama nueva pero en lugar de trabajar en el repositorio directamente trabajas en uno propio, lo que implica que puedes hacer lo que quieras con ello ya que no tocara el repositorio original. Tras hacer los cambios necesarios así como cuando hacíamos una rama diferente debíamos usar MERGE para fusionar los datos y seleccionar que se queda y que no se queda en el caso del FORK el creador del repositorio podrá examinar de antemano los cambios que has realizado y decidir si acepta esos cambios o no. En pocas palabras, es como trabajar sobre una copia de seguridad que no afecta a la versión original del repositorio.

Git utilizando Eclipse Neon

El Software Eclipse Neon es una herramienta gratuita para trabajar en JAVA, sin embargo es posible instalarle complementos para poder trabajar con Git, ademas de ello es mucho más fácil y cómodo para los usuarios que no están familiarizados a trabajar con comandos de consola/terminal ya que dispone de un entorno gráfico (UI = User Interface) que facilita mucho el trabajar más rápidamente y eficiente sin necesidad de escribir los comandos, pero a su vez puede ser un poco caótico a la hora de trabajar con diferentes proyectos a la vez o incluso dar conflictos y errores al juntarse la base de JAVA con el complemento de Git.
